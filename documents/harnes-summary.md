# ハーネスエンジニアリングとは？うまく設計するためのコツ

## 1. 結論：ハーネスエンジニアリングとは何か

**ハーネスエンジニアリングとは、「AIエージェントが安定して成果を出すための“環境そのもの”を設計する営み」である。**

LangChain の整理を借りれば、

> **Agent = Model + Harness**

であり、モデルの賢さだけでは成果は決まらない。モデルの「外側」── ルール、ツール、コンテキスト、検証ループ、運用のすべて ── を意図的に設計することが、現代のAI駆動開発の中心課題になっている。

PCに例えるとイメージしやすい：

| 構成要素 | 例え |
| --- | --- |
| モデル | CPU |
| コンテキストウィンドウ | RAM |
| エージェント | アプリケーション |
| **ハーネス** | **OS（アプリが動く土台）** |

**AIにとってハーネスはOSである。** どれだけCPU（モデル）が速くても、OS（ハーネス）が貧弱ならアプリは安定しない。

### 起源と定義の核心

最初にこの言葉を使ったのは Mitchell Hashimoto 氏とされる。彼の定義はシンプルだ：

> エージェントがミスをしたら、その都度、二度と同じミスをしないように仕組みを作り込め。

そのために必要なのは大きく2つ：

1. `AGENTS.md`（ルールファイル）の継続的改善
2. エージェント専用のツール整備（テストランナー、スクショ、検証スキル等）

つまりハーネスエンジニアリングの本質は、**「AIに毎回同じ説教をする」のではなく「同じ過ちを構造的に再発させない仕組みに落とし込む」こと**にある。

### なぜ今これが重要なのか

- OpenAI の Codex チームは、わずか3〜7名で5ヶ月間に約100万行・1500PR を「人間が手書きしたコード0行」で構築した。
- SWE-bench の分析では、**同じモデルでもハーネスを変えるだけでスコアが20点以上動く**ことが報告されている。フロンティアモデル同士の差より、ハーネスの差のほうが大きい。
- 進化の流れは「プロンプトエンジニアリング → コンテキストエンジニアリング → ハーネスエンジニアリング」と確実にレイヤーを上げてきている。

## 2. ハーネスの肝（本質的な要素）

各記事で繰り返し強調されるのは、次の3つのコア要素である。

### ① ルールファイル（行動規範）

`CLAUDE.md` / `AGENTS.md` / `.kiro/steering/*.md` のような、AI向けの指示書。

- 書くべきは「AIが間違えやすいプロジェクト固有のこと」だけ。一般的なベストプラクティスは書かない（既に知っている）。
- 百科事典ではなく **「目次（マップ）」** として設計する。OpenAI も大きな `AGENTS.md` を作って崩壊させ、約100行の入口ファイル＋構造化された `docs/` に分離することで成功した。
- 失敗するたびに1行ずつ追記して育てる、**生きたドキュメント**として扱う。

### ② フィードバックループ（自動検証と修正）

「実装 → 検証 → 修正」のループを、人間ではなく仕組みが回す状態をつくる。

```
1. AIが実装する
2. テスト/リンタ/型チェックが失敗を返す
3. 失敗理由をAIが読む
4. AIが修正する
5. 通るまで繰り返す
```

ここで決定的に重要なのは、**「失敗を、次の行動につながる信号に変える」**こと。
そのためにOpenAIは、カスタムリンタのエラーメッセージそのものを「エージェントがそのまま読んで直せる」形式で書いている。

### ③ コンテキスト管理（記憶の外部化）

エージェントのコンテキストウィンドウは有限で、会話をまたぐと記憶を失う。だから：

- 進捗・判断・設計を **外部ファイル（progress.md / TODO.md / exec-plans/）に書き出す**
- 必要なものだけを「選択」して渡し、不要なものを見せない
- 長い情報は「圧縮（要約）」する
- 関係ない情報が混ざらないよう、サブエージェントに「分離」する
- **Slackや人の頭の中の知識は、エージェントにとっては「存在しない」**。だからリポジトリに全部入れる

## 3. うまく設計するためのコツ（実践原則）

ここからが本題。各記事を通読して見えてきた、**ハーネスを設計する上での実践的な勘所**をまとめる。

### コツ1: 「Legibility（把握可能性）」をゴールにする

OpenAI が最重要視している概念。**「エージェントがシステムの挙動をどれだけ定量的に把握できるか」**である（`readability`＝可読性とは別物）。

具体策：
- UIなら Chrome DevTools MCP を埋め込み、DOM/スクショ/操作をスキル化する
- 監視ならworktreeごとにエフェメラルなo11yスタックを立て、Codex が LogQL/PromQL でクエリできるようにする
- これにより「起動を800ms以内に保証して」のような**振る舞いの指示が実行可能になる**

→ 「人間に見えるもの」ではなく **「エージェントが定量的に把握できるもの」**を増やすのが、設計のゴールである。

### コツ2: 不変条件は強制し、実装の細部は自由にさせる

**境界（boundaries）は厳格に、内部は自由に。** 優れた組織運営と同じ。

- 「境界でデータをパースせよ」は強制するが、「Zod を使え」とは指定しない
- レイヤードアーキテクチャや依存方向は、**カスタムリンタと構造テストで機械的に強制**する
- 命名規則・ファイルサイズ制限・構造化ロギング・プラットフォーム固有の信頼性要件もリンタに落とす
- 生成物が人間好みのスタイルでなくても、**正しく・保守可能で・把握可能なら気にしない**

通常は数百人規模になるまで後回しにする「やり過ぎな設計」が、エージェント協業では **最初から必要な前提条件**になる。

### コツ3: 深さ優先（depth-first）で進める

幅優先で全体像を浅く作るのではなく、**1つのユーザーストーリーを設計→実装→テスト→検証まで縦に貫く**。

その過程で生まれる「部品・抽象・仕組み」を再利用して次のストーリーに使う。
幅優先は初動が速いが、再利用資産が育たないため、長期的にはエージェントとの相性が悪い。

### コツ4: 知識基盤（system of record）をリポジトリに集約する

OpenAI の `docs/` 構成は強力なリファレンス：

```
AGENTS.md          ← 約100行のマップ（目次）
ARCHITECTURE.md
docs/
├── design-docs/    ← core-beliefs.md など、設計判断の軸
├── exec-plans/     ← 実行計画を第一級アーティファクトとして版管理
│   ├── active/
│   ├── completed/
│   └── tech-debt-tracker.md
├── product-specs/  ← 仕様の索引と本体
├── references/     ← 外部ライブラリのLLM向け資料
└── DESIGN.md / FRONTEND.md / RELIABILITY.md / SECURITY.md / ...
```

ポイント：
- **AGENTS.md は目次に徹する**、知識本体は `docs/` に置く
- 実行計画（exec-plans）も**バージョン管理されたアーティファクト**にする
- ドキュメント自体が腐らないよう、**専用リンタ/CI/エージェント**でリンク切れやドリフトを検知し自動PRを出す
- これにより **段階的開示（progressive disclosure）** が可能になる

### コツ5: 「ゴールデンプリンシプル」とGCで腐敗に抗う

エージェントは既存パターンを参照するため、**微妙なコードが混ざるとそのうち全体がドリフトする**（AI Slop問題）。

対策：
- 「黄金原則」をリポジトリにチェックインし、機械的にチェックできる形にする
  - 例：「場当たりヘルパーを作らず共有パッケージを使う」「型を推測せず境界で検証する」
- 定期実行のスコアリング・エージェントを走らせ、**小さな修正PRを高速に出してマージ**する
- これは **GC（ガベージコレクション）**であり、技術的負債を「複利で膨らむ前に毎日少しずつ返す」発想

### コツ6: 軽量に始め、段階的に育てる（3レベル）

**最初から壮大な仕組みを作らない。** AI駆動開発協会の整理が分かりやすい：

| レベル | やること | 状態 |
| --- | --- | --- |
| **Lv1: ルールを明文化** | AGENTS.md/CLAUDE.md を「マップ」として書く | ガイド |
| **Lv2: 制約と検証を仕組み化** | リンタ・型チェック・テスト・hookで実装→検証→修正ループ。安全ガードレール（本番アクセス禁止等）も。 | **ガードレール** |
| **Lv3: 観測と継続改善** | スループット/品質/人間負荷/安全性/ハーネス健全性の指標を見て、ハーネス自体を育てる | 自己進化 |

いきなりLv3から入らない。Lv1→Lv2→Lv3 の順番を守る。

### コツ7: 失敗前提で多層防御する

ルールファイル / 権限制御 / リンタ / テスト / レビューエージェント / 人間承認 ── これらは **どれか1つで完全に守るものではなく、重ねて効かせるもの**。
1つのガードが破られても次の層で止まる設計にする。

### コツ8: 人間を最後の承認者として残す

自律性を高めても人間を完全に外さない。とくに **本番変更・権限操作・顧客影響の大きい差分**は人間が最終承認する前提で設計する。

OpenAI のフローも「判断が必要な場合のみ人間にエスカレーションする」を組み込んでいる。

### コツ9: 日々のワークフローを “Explore → Plan → Implement → Verify → Record” に固定する

エージェントにいきなり「作って」と投げない。

1. **Explore** — 関連コード・仕様・構造を読む
2. **Plan** — 小さな作業単位に分ける
3. **Implement** — 1タスクずつ実装
4. **Verify** — テスト・lint・レビューで検証
5. **Record** — 進捗と判断を記録し次回に引き継ぐ

加えて **「1セッション1機能、1PR1目的」** の粒度を守る。

### コツ10: スループットが上がるとマージ哲学が変わることを受け入れる

OpenAI の経験：

- マージゲートは最小化される
- PRは短命になる
- フレーキーテストは原因究明より「再実行で通れば良し」とする場合もある
- **「実装は安く、待機は高い」**という損益分岐点の逆転が起きる

低スループット環境では無責任な振る舞いだが、高スループット環境では適切な妥協点になる。

### コツ11: リンタとテストへの投資が最大のレバレッジ

プロンプトは **非決定的かつ有限**だが、リンタ/テストは **決定的**である。

OpenAI が500万行規模にスケールできた本質は、**非決定的なAIコンテキスト層を薄く保ち、決定的な検証層を厚くした**ことにある。

→ 「不変条件を定量的に定義できること」こそがハーネスエンジニアリング時代の最重要スキル。

## 4. 避けるべき5つのアンチパターン

1. **作り込みすぎる** — 最初から複雑にすると、モデルや運用が変わった瞬間に壊れる
2. **ルールを書いて終わる** — 実行結果を見ずに放置すると、ハーネスはすぐ陳腐化する
3. **曖昧な指示に頼る** — 「いい感じにやって」は再現性ゼロ。禁止事項と期待結果は具体的に
4. **検証を人間に押しつける** — 機械で返せるものまで人間レビューに回すと、すぐに詰まる
5. **暗黙知を放置する** — 頭の中・Slack・Google Docs にしかない知識は、エージェントにとって存在しない

## 5. 注意したい論点（Fowler 氏の批判より）

Martin Fowler 氏は OpenAI の発信に対して次の点を指摘している。これは設計時のバランス感覚として覚えておきたい。

- OpenAIは **内部品質と保守性にしか言及していない**。プロダクトの機能性・振る舞いの検証についてはほぼ触れていない。
- 「コードはAIで保守可能」と主張するインセンティブが OpenAI にはある（ポジショントーク疑惑）。
- 大量のマークダウンを書くこと自体がハーネスではない。本質は **広範なツール、ナレッジベースのキュレーション、コードとリポジトリ設計そのもの**。
- **コード設計はそれ自体が巨大なコンテキスト**である。

→ ハーネスエンジニアリングは「リポジトリにmdを置くこと」ではない。**プロダクトの振る舞いを定量・自動・計測可能に寄せていく総合的な営み**であることを忘れない。

## 6. まとめ：1行でいうなら

> **ハーネスエンジニアリングとは、「エージェントが二度と同じミスをしないように、環境・ツール・ルール・検証ループを総合的に設計し、AIにとって把握可能（legible）な世界を作り続ける営み」である。**

設計の中心原則を5つに圧縮するなら：

1. **Legibility ファースト** — エージェントが定量的に把握できる世界を作る
2. **境界は厳格、内部は自由** — 不変条件を機械的に強制し、実装の細部は任せる
3. **決定的な検証層を厚く、非決定的なAI層を薄く** — リンタ・テスト・型に投資する
4. **軽量に始め、段階的に育てる** — Lv1ルール → Lv2検証 → Lv3観測の順
5. **人間を最後の承認者に残す** — 自律性は上げても、責任の最終地点は人間

最後に最も大事なこと：**AIは増幅器である**。良い土台を作れば良い成果が増幅され、ゴミを自動化すれば自動でゴミが量産される。だから先にハーネスへ投資する価値がある。
