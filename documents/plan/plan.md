# ハーネスエンジニアリング 実践計画書

## 0. この計画の目的

`harnes-summary.md` で整理した「ハーネスエンジニアリングのコツ」を、**実際に手を動かして体感する**ことが目的。
作るプロダクトそのものよりも、**プロダクトを作る過程で「ハーネス」を育てる体験**にフォーカスする。

> 主役はプロダクトではなく、ハーネス。プロダクトはハーネスを試すための題材。

---

## 1. 題材プロダクト（What to build）

**「Claude Code 使い方ダッシュボード」** ── Notion ライクに Markdown ファイルを閲覧できる、自分専用のローカルダッシュボード。

### 最低限の要件（あえてゆるく）

- ローカルの Markdown ファイル群を読み込んでブラウザで表示できる
- サイドバーで階層的にファイルを選択できる（Notion風のツリー）
- 本文エリアで Markdown がレンダリングされる
- 検索機能（タイトル・本文）
- 「Claude Code の使い方」というドメインに最適化した何らかの工夫（例：スラッシュコマンド一覧ページ、tips検索など）

### あえて決めないこと（ハーネスの中で決める）

- 技術スタック（Next.js / Vite+React / Astro / SvelteKit など）
- スタイリング手法
- ディレクトリ構造
- ストレージ方式（ファイル直読み / SQLite インデックス / etc）
- デプロイ形態（ローカル起動のみ想定）

→ これらは「エージェントに深さ優先で1機能を貫通させる過程」で決めていく。

---

## 2. なぜこの題材がハーネス練習に向いているか

| 要素 | 適している理由 |
| --- | --- |
| **UI を持つ** | Chrome DevTools MCP による検証ループが活きる |
| **Markdown 処理** | テスト可能な純粋ロジックが多く、リンタ/テストの効果を体感できる |
| **個人用** | 仕様の正解が自分の中にあり、要件変更を恐れず壊せる |
| **小さい** | ハーネスを育てる時間を確保できる |
| **ドッグフーディング可能** | 完成後、自分のClaude Code学習にそのまま使える |

---

## 3. 構築するハーネスの全体像

`harnes-summary.md` の「3レベル」に沿って、以下を段階的に構築する。
**いきなり全部作らない。Lv1 → Lv2 → Lv3 の順で進める。**

### Lv1: ルールを明文化する（最初の1日）

#### 作るもの

- **`AGENTS.md`（または `CLAUDE.md`）** ── 100行以内のマップ。以下を含む：
  - このプロジェクトは何か（1段落）
  - どこを触ってよい / 触ってはいけないか
  - ビルド・起動・テスト・lint コマンド
  - `docs/` への入口リンク
- **`docs/` ディレクトリ** ── 知識基盤（最小構成）
  - `docs/ARCHITECTURE.md` ── ディレクトリと依存方向（最初は空でOK、決まり次第追記）
  - `docs/exec-plans/active/` ── 進行中の実行計画
  - `docs/exec-plans/completed/` ── 完了済み
  - `docs/product-specs/` ── 自分用の仕様メモ
  - `docs/core-beliefs.md` ── 設計判断の軸（後述の「黄金原則」もここに育てる）

#### このフェーズの設計判断

- AGENTS.md は「百科事典ではなく目次」に徹する
- 技術スタックすら最初は決め打ちしない（Codex/Claudeに第1歩を出させて、それを正とする）
- 失敗するたびに `core-beliefs.md` か `AGENTS.md` に1行ずつ追記して育てる

---

### Lv2: 制約と検証を仕組みに埋め込む（核心フェーズ）

ここがハーネスエンジニアリング体験の **本丸**。
3つのフィードバックループ＋カスタムサブエージェントを並行して構築する。

#### ループ① 静的検証ループ（決定的・速い）

| 種類 | ツール候補 | 目的 |
| --- | --- | --- |
| Linter | ESLint / Biome | コーディング規約違反を機械的に止める |
| Formatter | Prettier / Biome | スタイル差分を消す |
| 型チェック | TypeScript `tsc --noEmit` | 型エラーを止める |
| カスタムLint | 自作スクリプト or eslint-plugin | プロジェクト固有の不変条件を強制 |

**重要**: Claude/Codex がそのまま読んで直せるよう、**カスタムLintのエラーメッセージはエージェントフレンドリーに書く**。
例：`"Use the shared <Markdown> component from src/ui/markdown instead of importing 'react-markdown' directly. See docs/core-beliefs.md#markdown-rendering"`

#### ループ② 単体・統合テストループ

- Vitest / Jest などで、Markdown パース・検索・ファイルツリー構築のテストを書く
- **テスト失敗時の出力を、エージェントが「次に何をすべきか」読み取れる粒度にする**
- AI に「テストが通るまで自律ループ」させる前提で、テスト名と assertion メッセージを意味的にする

#### ループ③ E2E / UI 検証ループ（Chrome DevTools MCP の出番）

これが今回の **目玉**。

**やること:**
1. Chrome DevTools MCP を Claude Code に接続する
2. 開発サーバーを起動した状態で、Claude に以下のスキルを使わせる：
   - ページを開く / ナビゲートする
   - DOM スナップショットを取得する
   - スクリーンショットを撮る
   - コンソールエラー・ネットワークエラーを確認する
   - クリック・入力などのインタラクションを行う
3. Claude が UI 変更を加えた後、**自分でブラウザを開いて検証→失敗を検出→修正**を回せる状態にする

**設計のコツ（summary.md より）:**
- worktree やブランチごとに分離したインスタンスで動かせると理想（最初は単一インスタンスでも可）
- 「起動が遅い」「コンソールエラーが0であること」のような **振る舞いを定量的な指示**にできる状態をゴールにする
- スナップショットを `docs/exec-plans/active/` に保存し、進捗ログとして残す

#### ループ④ レビュー用サブエージェント（コンテキスト分離）

- Claude Code の `.claude/agents/*.md` にカスタムサブエージェントを定義する
- 役割: メインのコーディングエージェントが書いたコードを **別コンテキスト** でレビューする
  - これは OpenAI の Ralph Wiggum ループの最小再現
- 想定する初期エージェント:
  - **`code-reviewer`**: 差分を読んで黄金原則違反・型の甘さ・テスト不足を指摘する
  - （余裕があれば）**`spec-checker`**: 実装が `docs/product-specs/` の意図とズレていないか確認する
- ポイント:
  - サブエージェントには **狭い責務** を与える（万能レビュアーにしない）
  - 出力は「指摘 → 直す手順」がメインエージェントにそのまま渡せる形式にする

#### 安全ガードレール

- 編集可能ディレクトリを `src/` `docs/` `tests/` に限定（ルートの設定ファイルは原則人間承認）
- `.env` や認証情報は触らせない
- 危険コマンド（`rm -rf` 系、`git push --force` 等）は明示的に禁止して `AGENTS.md` に書く
- Pre-commit hook で lint/format/test を走らせ、通らないコミットを止める

---

### Lv3: 観測と継続改善（後半フェーズ）

#### 育てるもの

- **黄金原則（golden principles）** を `docs/core-beliefs.md` に蓄積する
  - 例：「Markdownレンダリングは共有コンポーネント経由」「ファイルアクセスは src/io/ 配下のみ」
- **GCタスク** ── 週1で「ドリフトしていないか」をチェックさせる Claude タスクを定義する（手動でもOK）
- **指標を見る** ── 最低限以下を記録：
  - ループ1周あたりの自己修正回数（少ないほどハーネスが効いている）
  - 人間が介入した回数（理想は単調減少）
  - 同じ種類のミスの再発回数（黄金原則化のサイン）

#### Lv3 でさらに育てるサブエージェント

`.claude/agents/` にスコープを絞った専門エージェントを追加していく：

- **`golden-principle-checker`**: `docs/core-beliefs.md` の黄金原則違反を検出し、修正PRを出す GC エージェント
- **`docs-drift-detector`**: 実装と `docs/ARCHITECTURE.md` / `product-specs/` の乖離を検知する
- **`exec-plan-archiver`**: `active/` の完了済み計画を `completed/` に移し、学びを `core-beliefs.md` に追記する

これらは**手動起動でも構わない**。重要なのは「人間がレビューで指摘していたこと」を1つずつエージェント化していく営みそのもの。

## 4. 開発ワークフロー

`harnes-summary.md` の **Explore → Plan → Implement → Verify → Record** を厳守する。

```
[Explore]   関連コード・docs/ を読む
   ↓
[Plan]      docs/exec-plans/active/ に1機能分の実行計画を書く
   ↓
[Implement] 1セッション1機能の粒度で実装
   ↓
[Verify]    ループ①②③を全部通す
   ↓
[Record]    完了したら exec-plans/completed/ へ移動、学びを core-beliefs.md に追記
```

**1セッション1機能、1PR1目的を死守する。**

## 5. ステップバイステップ実行計画

### Phase 0: 準備（半日）

- [ ] `documents/` の外側にプロジェクトディレクトリを切る（例：`harnes/dashboard-app/`）
- [ ] git 初期化、最小の `AGENTS.md` を Claude に書かせる
- [ ] Chrome DevTools MCP の接続を確認する（疎通テストだけ）
- [ ] `docs/` 雛形を作る

### Phase 1: Lv1 ハーネス + 最初の縦串（1〜2セッション）

- [ ] Claude に「Markdownファイルを1つだけ表示するページ」を作らせる（深さ優先の最初の1本）
- [ ] このときに必要な最小ツール（dev server, build）を確立
- [ ] 失敗したら、**コードを直す前にハーネスを直す** ── AGENTS.md に1行追記する習慣をつける

### Phase 2: Lv2 ループ構築（複数セッション）

- [ ] Linter / Formatter / 型チェックを導入
- [ ] 最初のテストを書く（Markdown パーサ周り）
- [ ] **Chrome DevTools MCP 経由の検証スキルを Claude に試させる**
  - 例：「dev server を起動して、`/` を開いて、h1 タグのテキストをスナップショットして」
- [ ] Pre-commit hook を導入
- [ ] カスタム lint を1個だけ書く（プロジェクト固有の禁止事項）
- [ ] **`.claude/agents/code-reviewer.md` を作る** — 最初のカスタムサブエージェント（差分レビュー専用）

### Phase 3: 機能を増やしながらハーネスを育てる

- [ ] サイドバーツリー → テスト + DevTools MCPでの目視確認
- [ ] 検索機能 → 単体テスト + DevTools MCPでのインタラクション検証
- [ ] 同じミスが2回出たら → 黄金原則化 or リンタ化
- [ ] 必要に応じて2つ目のサブエージェント（例: `spec-checker`）を追加

### Phase 4: Lv3 観測（任意）

- [ ] ハーネス健全性チェック用の Claude プロンプトを作る
- [ ] 定期的に core-beliefs.md を見直す
- [ ] **GC系サブエージェント** を `.claude/agents/` に追加（`golden-principle-checker` / `docs-drift-detector` など）

## 6. 成功基準（このプロジェクトのDoD）

**プロダクトの完成度ではなく、ハーネスとしての成熟度で測る。**

1. **`AGENTS.md` が「目次」として機能している** ── 100行以内、詳細は docs/ に分離されている
2. **Chrome DevTools MCP を使った UI フィードバックループが少なくとも1回回った** ── 「Claudeが自分でブラウザ開いて、エラー見つけて、自分で直した」が1回でも起きれば成功
3. **テスト/Lint が壊れた状態で commit できない** ── 機械的ガードレールが効いている
4. **同じミスを2回しないことが少なくとも1件、ルール化されている** ── Hashimoto 流の本質を体験している
5. **`docs/exec-plans/completed/` に最低3つの実行計画が積まれている** ── 計画→実行→記録のループが回った証拠
6. **`.claude/agents/` にカスタムサブエージェントが少なくとも1つ存在し、実際にレビューループで使われた** ── コンテキスト分離による多エージェント協調を体験している

## 7. 意識すべきアンチパターン（自戒）

- ❌ いきなり Lv3 から始めて壮大な仕組みを作ろうとする
- ❌ AGENTS.md に「あれもこれも」書いて百科事典化させる
- ❌ AI の出力を目で見て手で直す（フィードバックループの仕組み化を怠る）
- ❌ Chrome DevTools MCP を「繋いだだけ」で終わらせ、実際の検証ループに使わない
- ❌ プロダクトの完成を急ぎ、ハーネスへの投資をサボる
